# Аудит эмиссии ∞ и внедрение emission-guard

**Дата:** 2026-07-23
**Область:** все серверные пути, начисляющие ∞ (`wallets.timecoin`) без прямого 1:1 обеспечения реальным TC в казначействе Solana.
**Продолжение:** [docs/solana-reserve-audit.md](./solana-reserve-audit.md) (риск Р4), задача из `docs/roadmap.md`.

---

## TL;DR

- Найдено **6 источников эмиссии ∞**, не обеспеченных реальным TC в казначействе (см. таблицу ниже). Всего 4 других начисления/перевода в коде — это wash-операции (перекладывание уже существующего ∞ между пользователями/представлениями), эмиссией не являются.
- Добавлен единый контрольный модуль `backend/src/lib/emission-guard.ts` (`canEmitUnbacked(amount)`), который сверяет, что после начисления `amount` суммарный ∞ в обращении (`SUM(wallets.timecoin)`) всё ещё не превысит баланс казначейства TC на Solana. Если превысит — начисление отклоняется.
- Guard подключён ко всем 6 источникам. **Уже начисленные пользователям балансы нигде не уменьшаются и не трогаются** — ограничение действует только на будущий, ещё не совершённый инкремент.
- Guard **fail-closed**: если баланс казначейства невозможно проверить (Solana не сконфигурирован, RPC недоступен и т.п.), начисление блокируется — риск превышения резерва хуже, чем временно недоступный бонус.
- Для каждого источника выбрана деградация, которая не ломает основной пользовательский флоу (см. раздел 3): бонусы остаются доступны позже, промокод не сгорает при отказе, тело стейка возвращается полностью в любом случае, tc-market buy просто частично исполняется через стакан.

## 1. Механизм: `canEmitUnbacked()`

`backend/src/lib/emission-guard.ts`:

```ts
export async function canEmitUnbacked(amount: number): Promise<boolean> {
  if (!(amount > 0)) return true

  try {
    const treasuryTc = await solanaService.getTreasuryBalance()
    const row = db.prepare(`SELECT COALESCE(SUM(timecoin), 0) as total FROM wallets`).get() as { total: number }
    return row.total + amount <= treasuryTc
  } catch (err) {
    captureError("[emission-guard] reserve check failed, blocking unbacked emission:", err)
    return false
  }
}
```

Логика идентична уже существующему публичному эндпоинту `GET /api/tc/reserve-status` (казна vs. `SUM(timecoin)`), но выполняется как прямой запрос внутри бэкенда (не через HTTP) непосредственно перед конкретным начислением, чтобы решение принималось на основе как можно более свежих данных и без сетевого прыжка через собственный API.

Проверка выполняется **до** открытия соответствующей БД-транзакции — `SolanaService.getTreasuryBalance()` асинхронный (сетевой вызов к Solana, с 5-секундным кэшем), а используемый в проекте синхронный `node:sqlite` API не даёт безопасно делать `await` посреди открытой транзакции.

## 2. Источники эмиссии ∞ — до и после

| # | Источник | Файл | Величина | Обеспечение | Guard |
|---|---|---|---|---|---|
| 1 | Реферальный бонус | `auth.controller.ts` → `register()` | +10 ∞ за каждого приглашённого, без лимита на кол-во | Нет | ✅ |
| 2 | Демо-бонус за конвертацию | `demo.routes.ts` → `/convert` | +50 ∞ разово (флаг `demo_bonus_claimed`) | Нет | ✅ |
| 3 | Промокоды (`type=timecoin`) | `promo.routes.ts` → `/redeem` | Произвольная сумма (задаёт админ) | Нет | ✅ |
| 4 | Стейкинг — проценты | `stakes.routes.ts` → `/:id/unstake` | 3–24% годовых от тела стейка | Нет (само тело стейка — да, это уже принадлежащие пользователю ∞) | ✅ (только на проценты) |
| 5 | Награда за фидбек | `feedback.routes.ts` | +5 ∞/день, лимит 1/день/юзер | Нет | ✅ |
| 6 | TC-market «эмиссия» (buy-fallback) | `tcmarket.routes.ts` → `/buy` | Минт остатка заявки при нехватке asks в стакане | Нет (обеспечено только внутренним `cash_usd`, не реальным TC) | ✅ |

Ни у одного из 6 источников не было верхнего предела, завязанного на реальный резерв, до этого изменения — только точечные защиты от повторного фарма одного и того же бонуса (флаги/дневные лимиты), не связанные с фактическим покрытием резерва.

### Не являются источниками эмиссии (проверено, не тронуто)

| Путь | Файл | Почему не эмиссия |
|---|---|---|
| `/wallet/transfer` | `wallet.routes.ts` | Перевод ∞ между существующими кошельками — сумма ∞ в системе не меняется |
| `/wallet/convert` | `wallet.routes.ts` | Конвертация между внутренними валютами (credits/shards/crystals ↔ ∞ и т.п.), не создаёт новый ∞ из воздуха отдельно от уже существующего баланса |
| Аренда twin (`twin.routes.ts`) | `twin.routes.ts` | Перевод ∞ между арендатором и владельцем |
| `matching-engine.ts` → `executeTrade()` | `matching-engine.ts` | Исполнение сделки на внутреннем ордербуке — ∞ переходит от покупателя к продавцу, не создаётся |
| `/tc-market/sell` burn-fallback | `tcmarket.routes.ts` | Сжигание (`burn`) ∞ при нехватке bids — уменьшает объём ∞ в обращении, обратное эмиссии, guard не нужен |
| `POST /api/tc/deposit` | `tc.routes.ts` | Обеспечен 1:1 реальным TC, физически внесённым пользователем в казначейство (верифицируется on-chain) |

## 3. Как деградирует каждый источник при отказе guard'а

Цель — не ломать основной продуктовый флоу целиком, только капать будущий необеспеченный прирост ∞:

- **Реферальный бонус**: реферал всё равно засчитывается (`INSERT INTO referrals ... status='reserve_capped'`), просто без начисления +10 ∞ рефереру. Можно позже пересчитать вручную, если потребуется, когда резерв восстановится — история не теряется.
- **Демо-бонус**: флаг `demo_bonus_claimed` не выставляется при отказе → пользователь сможет получить бонус позже, при повторном вызове `/demo/convert` (или отдельным ретраем), когда резерв снова покрывает эмиссию.
- **Промокод**: при отказе `promo_redemptions` не создаётся и счётчик `uses` не увеличивается — код остаётся полностью неиспользованным для этого пользователя, можно применить позже.
- **Стейкинг**: тело стейка (`stake.amount_tc`) возвращается пользователю **всегда и полностью** — это его собственные ранее застейканные ∞, не эмиссия. Капается только начисленный процент (`reward`), причём капается до нуля целиком, а не частично — сумма процентов обычно мала относительно тела стейка, частичное урезание не даёт ощутимой точности ценой усложнения кода.
- **Фидбек-бонус**: `rewardGranted: false` в ответе API, фидбек всё равно сохраняется и уведомления отправляются — только TC не начисляется.
- **TC-market buy**: заявка исполняется частично — на объём, который смог закрыть встречный стакан (`orderbookExecutedTc`); недостающий объём (который раньше безусловно минтился) просто не покупается через эмиссию. Существующая проверка `if (totalTc <= EPS)` уже корректно обрабатывает случай, когда и стакан, и минт дали ноль.

## 4. Проверка

- `npx tsc --noEmit` в `backend/` — без ошибок (в т.ч. после перевода `stakes.routes.ts` `/:id/unstake` и `tcmarket.routes.ts` `/buy` с синхронных на асинхронные обработчики).
- Ручной построчный обзор всех 10 найденных мест начисления/списания ∞ в `backend/src` — подтверждена классификация 6 эмиссионных vs. 4 wash-путей (раздел 2).

## 5. Не сделано намеренно (требует отдельного продуктового решения)

- **Верхний предел на сам стейкинг-APR или частичное капание процентов** — сейчас при недостатке резерва проценты капаются полностью до нуля, а не пропорционально урезаются. Пропорциональное урезание потребовало бы дополнительной логики распределения дефицита резерва между конкурирующими операциями (риск гонки при параллельных unstake), что выходит за рамки этого аудита.
- **Фронтендная константа `TC_TOTAL_CAP`** (`lib/tc-market.ts`, риск Р4 в `solana-reserve-audit.md`) по-прежнему не проверяется на бэкенде — она про общий лимит эмиссии tc-market, а не про обеспеченность резервом, это отдельная задача.
- **Уведомление пользователя о причине отказа** (например, в UI показать «бонус временно недоступен, казна перегружена» вместо молчаливого `rewardGranted: false`) — текстовые доработки фронтенда не входили в объём этой серверной задачи.
