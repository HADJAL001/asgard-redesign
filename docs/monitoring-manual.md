# Инструкция по ручному мониторингу продакшена (OSGARD backend)

Автоматика (`keep-alive.yml` + `daily-status-report.yml`) покрывает базовый uptime-мониторинг, но раз в день стоит проверять состояние сервера руками — автоматика не ловит всё (деградацию логики, рост ошибок в бизнес-эндпоинтах, аномалии в данных).

## Что проверять каждое утро (5–10 минут)

1. **Telegram-канал алертов.**
   Если ночью пришло сообщение 🔴 (падение health/tc-market/state) или ⚠️ (медленный ответ >2с) — разобраться в первую очередь, см. раздел «Как реагировать» ниже.

2. **[docs/monitoring-dashboard.md](./monitoring-dashboard.md).**
   Обновляется ежедневно в 08:00 МСК. Смотреть:
   - Uptime за 7 дней — если ниже ~99%, искать причину (см. лог `monitoring/status-log.csv` в ветке `monitoring-data`, там видно конкретные метки времени сбоев).
   - Среднее время ответа — если растёт день ото дня, это признак деградации (нагрузка на БД, утечка памяти, холодный старт).
   - Количество ошибок 5xx за неделю.

3. **Вкладка GitHub Actions.**
   `Actions → keep-alive` — проверить, что последние запуски (раз в 10 минут) зелёные. Если workflow вообще не запускается (нет новых прогонов) — это отдельная проблема (не с бэкендом, а с CI): проверить, не отключён ли workflow, не истёк ли `GITHUB_TOKEN`/permissions.
   `Actions → daily-status-report` — что вчерашний отчёт отработал успешно и дашборд обновился (коммит `chore(monitoring): daily status report ...` в `main`).

4. **Ручной smoke-тест (если есть сомнения в автоматике):**
   ```bash
   curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" https://osgardnewworld.com/api/health
   curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" https://osgardnewworld.com/api/tc-market/state
   ```
   Ожидается `200` и время ответа до ~1–2 секунд.

5. **Sentry** (бэкенд подключён через `@sentry/node`, см. `backend/src/lib/sentry.ts` и `Sentry.setupExpressErrorHandler` в `backend/src/server.ts`).
   Зайти в проект Sentry и посмотреть новые issue за ночь — там осядут необработанные исключения, даже если health-check при этом был зелёным (например, ошибка в конкретном эндпоинте под нагрузкой).

## Как реагировать на ошибки

| Сигнал | Вероятная причина | Что делать |
|---|---|---|
| 🔴 `/api/health` не отвечает | Сервер упал / не поднялся после деплоя / OOM | Зайти на хостинг, посмотреть процесс/под, при необходимости перезапустить; проверить последний деплой (не он ли уронил сервер) |
| 🔴 только `/api/tc-market/state` не отвечает, health зелёный | Проблема конкретно в matching-engine / БД-запросах этого роута | Смотреть логи хостинга по этому запросу + Sentry issues, проверить состояние таблицы `tc_market_state` (не заблокирована ли БД) |
| ⚠️ медленный ответ (>2с) единичный | Кратковременный пик нагрузки, холодный старт после простоя | Если единично и не повторяется — не паниковать, просто зафиксировать; если повторяется несколько раз подряд — см. следующий пункт |
| ⚠️ медленный ответ систематически (много раз за день) | Деградация (нагрузка на БД, утечка памяти, недостаточно ресурсов инстанса) | Смотреть график среднего времени ответа в дашборде за несколько дней, при устойчивом росте — эскалировать на апгрейд инстанса/оптимизацию запросов |
| Рост ошибок 5xx в дашборде без явного падения health | Ошибки в отдельных запросах (500 из-за необработанного исключения) | Смотреть Sentry — там будет стектрейс конкретной ошибки |
| `keep-alive`/`daily-status-report` не запускаются вообще | Проблема в CI, не в проде | Проверить `Settings → Actions` репозитория, что Actions включены и cron не отключён GitHub из-за неактивности репозитория (>60 дней без коммитов отключает scheduled workflows — коммитить хотя бы раз в 1–2 месяца) |

Если требуется временно отключить алерты (например, на время планового обслуживания) — не удалять secrets, а временно поставить репозиторий/workflow на паузу (`Actions → keep-alive → Disable workflow`), не забыть включить обратно.

## Куда смотреть логи

- **CI/健康-чеки:** `Actions → keep-alive` и `Actions → daily-status-report` в GitHub — вывод каждого шага, включая `::warning`/`::error` аннотации.
- **Сырые метрики uptime:** ветка `monitoring-data`, файл `monitoring/status-log.csv` (timestamp, endpoint, http_code, response_time_s, ok) — по нему можно быстро посчитать что угодно вручную (`awk`/Excel/Google Sheets).
- **Ошибки приложения (стектрейсы):** Sentry-проект бэкенда (см. `SENTRY_DSN` в переменных окружения бэкенда, `backend/src/lib/sentry.ts`).
- **Логи самого процесса/хостинга** (stdout/stderr Node-процесса, рестарты, OOM): зависит от того, где сейчас развёрнут `backend` (в репозитории нет закоммиченного `render.yaml`/`Procfile`/systemd-юнита на момент написания этой инструкции) — уточнить у того, кто настраивал деплой, и при первой возможности задокументировать здесь точный адрес консоли/дашборда хостинга и как туда заходить.
